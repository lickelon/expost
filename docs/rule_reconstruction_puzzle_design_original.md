# 정답 공개형 규칙 복원 퍼즐 게임 기획서

## 1. 한 줄 소개

> 최종 보드는 이미 완성되어 있다.  
> 플레이어는 답을 맞히는 것이 아니라, 그 답을 만들어낸 색상자들의 규칙을 복원한다.

---

## 2. 핵심 컨셉

이 게임은 일반적인 퍼즐게임처럼 숨겨진 정답을 찾는 구조가 아니다.

플레이어에게는 다음 정보가 처음부터 주어진다.

- 그리드 위에 놓인 여러 색 상자
- 상자들의 작용으로 만들어진 최종 상태
- 각 색 상자가 가질 수 있는 규칙 후보

플레이어가 해야 하는 일은 최종 상태를 직접 수정하는 것이 아니라,  
각 색 상자가 어떤 규칙으로 주변 칸에 영향을 주었는지 추론하고 조립하는 것이다.

즉, 이 게임의 정답은 완성된 보드가 아니라 **완성된 보드를 만들어내는 규칙 세트**다.

---

## 3. 장르 정의

### 기본 장르

- 퍼즐
- 규칙 추론
- 역공학 퍼즐
- 생성 규칙 복원 퍼즐

### 기존 규칙 추론 퍼즐과의 차이

기존의 규칙 추론 퍼즐은 보통 다음 구조를 가진다.

```text
규칙을 추론한다
→ 그 규칙을 이용해서 새로운 퍼즐을 푼다
```

이 게임의 구조는 다르다.

```text
최종 결과가 주어진다
→ 그 결과를 만든 규칙을 복원한다
```

따라서 규칙은 풀이 도구가 아니라, 그 자체가 플레이어가 제출해야 하는 답이다.

---

## 4. 기본 게임 루프

```text
1. 스테이지 입장
2. 초기 상자 배치 확인
3. 목표 최종 상태 확인
4. 색상자별 규칙 조립
5. 시뮬레이션 실행
6. 플레이어 결과와 목표 결과 비교
7. 일치하면 클리어
8. 더 단순한 규칙, 더 높은 평가를 위해 재도전 가능
```

## 5. 화면 구성

초기 MVP에서는 한 화면에 세 가지 정보를 동시에 보여주는 방식이 적합하다.

```text
[초기 보드]   [내 규칙 적용 결과]   [목표 보드]
```

### 초기 보드

- 색 상자 위치 표시
- 빈 칸 표시
- 장애물 또는 벽이 있다면 함께 표시

### 내 규칙 적용 결과

- 플레이어가 선택한 규칙을 적용한 결과
- 시뮬레이션 버튼을 누르거나 규칙 변경 시 자동 갱신
- 목표와 다른 칸은 강조 표시

### 목표 보드

- 스테이지의 최종 상태
- 플레이어가 직접 수정할 수 없음

### 하단 UI

- 색상별 규칙 선택 패널
- Submit / Simulate 버튼
- 틀린 칸 개수 표시
- 클리어 여부 표시

## 6. 플레이어 입력 방식

플레이어는 보드의 칸을 직접 칠하거나 수정하지 않는다.  
대신 색상자별 규칙을 부품처럼 조립한다.

예시:

```text
빨강 상자:
방향 = 상하좌우
거리 = 1칸
효과 = 숫자 +1

파랑 상자:
방향 = 대각선
거리 = 벽까지
효과 = 색칠
```

이 방식은 코딩 퍼즐처럼 보일 수 있는 요소를 줄이고,  
퍼즐게임으로서 직관성을 유지할 수 있다.

## 8. MVP 규칙 범위

처음부터 모든 규칙을 넣지 않는다.  
첫 데모에서는 다음 정도가 적절하다.

```text
보드 크기: 5x5
색상 수: 2개
방향: 상하좌우 / 대각선 / 가로 / 세로
거리: 1칸 / 2칸
효과: 숫자 +1
충돌 처리: 숫자 누적
스테이지 수: 10개
```

초기에는 색칠보다 숫자 누적이 더 적합하다.

이유:

- 충돌 처리가 명확함
- 여러 상자의 영향이 겹쳤는지 쉽게 보임
- 목표 결과와 비교하기 쉬움
- 플레이어가 원인을 분해하기 좋음

## 10. 유일해에 대한 방향

이 게임은 반드시 유일해일 필요가 없다.

중요한 것은 제작자가 의도한 단 하나의 규칙을 맞히는 것이 아니라,  
제시된 최종 상태를 실제로 만들어낼 수 있는 규칙 세트를 구성하는 것이다.

따라서 다음 구조가 적합하다.

```text
클리어 조건:
목표 보드와 동일한 결과를 만드는 규칙 세트 제출

추가 평가:
더 적은 규칙 항목
더 단순한 규칙
예외 조건이 적은 규칙
더 많은 테스트 보드에 적용 가능한 규칙
```

유일해를 강제하지 않으면 스테이지 제작이 쉬워지고,  
플레이어의 예상 밖 풀이도 인정할 수 있다.

단, 좌표 직접 지정과 같은 치트성 규칙은 막아야 한다.

비허용 예시:

```text
3행 5열을 칠한다
정답 패턴을 그대로 복사한다
특정 좌표 목록을 직접 입력한다
```

허용 예시:

```text
상자 기준 상하좌우 1칸
벽을 만날 때까지
가장 가까운 빈칸
영향받은 횟수만큼 숫자 증가
```

## 11. 난이도 조절 방식

이 게임의 난이도는 다음 요소로 조절할 수 있다.

### 11.1 색상 수 증가

```text
초반: 색 1개
중반: 색 2~3개
후반: 색 4개 이상
```

색이 늘어나면 어떤 결과가 어떤 색에서 비롯되었는지 분해해야 하므로 난이도가 올라간다.

### 11.2 규칙 항목 추가

초반에는 방향만 추론하게 하고,  
이후 거리, 효과, 조건, 충돌 처리 등을 순차적으로 추가한다.

추천 해금 순서:

```text
1장: 방향
2장: 거리
3장: 숫자와 색칠
4장: 벽과 장애물
5장: 겹침과 충돌 처리
6장: 적용 순서
7장: 조건부 규칙
8장: 규칙끼리 규칙을 바꾸는 후반 구조
```

### 11.3 보드 크기 증가

```text
4x4: 튜토리얼
5x5: 기본
6x6: 일반 스테이지
7x7 이상: 후반 또는 챌린지
```

다만 보드가 커진다고 무조건 좋은 것은 아니다.  
이 게임은 작은 보드에서 증거가 명확한 스테이지가 더 좋을 수 있다.

### 11.4 영향 영역 겹침

쉬운 스테이지:

```text
각 색의 영향이 거의 분리되어 있음
```

어려운 스테이지:

```text
여러 색의 영향이 같은 칸에 겹침
겹친 결과가 숫자로 합쳐짐
벽 때문에 일부 방향이 막힘
```

### 11.5 여러 테스트 보드

후반부에는 하나의 규칙 세트를 여러 보드에 적용하게 만들 수 있다.

```text
같은 빨강 규칙
같은 파랑 규칙
서로 다른 초기 배치
서로 다른 목표 결과
```

이 방식은 유일해 문제를 줄이고,  
플레이어가 더 일반적인 규칙을 찾도록 만든다.

## 12. 스테이지 제작 방식

이 게임은 스테이지 제작이 비교적 쉬운 편이다.

일반 퍼즐게임은 제작자가 직접 풀이 과정을 설계해야 하지만,  
이 게임은 다음 방식으로 만들 수 있다.

```text
1. 숨겨진 색상별 규칙을 정한다
2. 상자를 배치한다
3. 시뮬레이션을 돌린다
4. 생성된 결과를 목표 보드로 사용한다
5. 플레이어에게 초기 배치와 목표 보드만 보여준다
```

즉, 제작자는 최종 상태를 손으로 직접 만들 필요가 없다.  
규칙과 초기 배치를 만들면 최종 상태는 자동 생성된다.

### 좋은 스테이지 조건

좋은 스테이지는 단순히 결과가 예쁜 스테이지가 아니다.

```text
비슷한 가짜 규칙들을 탈락시킬 수 있는 증거가 있는 스테이지
플레이어가 원인을 분해할 수 있는 스테이지
틀렸을 때 왜 틀렸는지 비교가 가능한 스테이지
규칙 하나를 바꾸면 결과가 명확하게 달라지는 스테이지
```

## 13. Unity 구현 방향

### 13.1 핵심 원칙

Unity 오브젝트로 규칙을 구현하지 않는다.  
게임 로직은 순수 C# 데이터 시뮬레이터로 만들고,  
Unity는 그 결과를 화면에 보여주는 역할을 맡는다.

```text
StageData
RuleSet
Simulator
Validator
GridView
```

이렇게 분리하면 규칙 추가, 테스트, 스테이지 검증이 쉬워진다.

## 14. 주요 데이터 구조

### 14.1 StageData

```csharp
[CreateAssetMenu(menuName = "Puzzle/Stage")]
public class StageData : ScriptableObject
{
    public int width;
    public int height;
    public List<SourceBoxData> sources;
    public List<CellData> targetCells;
    public List<ColorRuleOptions> availableRules;
}
```

### 14.2 SourceBoxData

```csharp
[System.Serializable]
public class SourceBoxData
{
    public int x;
    public int y;
    public BoxColor color;
}
```

### 14.3 Rule

```csharp
public enum DirectionType
{
    Cross,
    Diagonal,
    Horizontal,
    Vertical,
    AllAround
}

public enum RangeType
{
    One,
    Two,
    UntilWall
}

public enum EffectType
{
    AddNumber,
    Paint,
    Erase
}

public class Rule
{
    public DirectionType Direction;
    public RangeType Range;
    public EffectType Effect;
}
```

### 14.4 RuleSimulator

```csharp
public class RuleSimulator
{
    public BoardState Simulate(StageData stage, Dictionary<BoxColor, Rule> rules)
    {
        var result = new BoardState(stage.width, stage.height);

        foreach (var source in stage.sources)
        {
            if (!rules.TryGetValue(source.color, out var rule))
                continue;

            ApplyRule(result, stage, source, rule);
        }

        return result;
    }
}
```

### 14.5 ApplyRule 분리

```text
GetAffectedPositions()
ApplyEffect()
ResolveConflict()
```

이 세 단위로 나누면 새로운 규칙을 추가하기 쉽다.

## 17. MVP 목표

첫 데모는 다음 정도로 제한한다.

```text
플랫폼: Unity PC 또는 WebGL
보드: 5x5
스테이지: 10개
색상: 1~2개
규칙 항목: 방향 / 거리 / 효과
효과: 숫자 +1 중심
충돌: 숫자 누적
UI: 초기 보드, 결과 보드, 목표 보드, 규칙 선택 패널
```

### MVP에서 확인해야 할 것

```text
최종 상태를 보고 규칙을 복원하는 행위가 재미있는가
규칙을 바꿀 때 결과가 변하는 과정이 직관적인가
틀렸을 때 왜 틀렸는지 이해 가능한가
스테이지를 빠르게 만들 수 있는가
유일해가 아니어도 플레이어가 납득하는가
```

## 18. 확장 아이디어

### 18.1 랭크 시스템

```text
C: 목표 결과와 일치
B: 규칙 수 제한 만족
A: 예외 조건 없이 클리어
S: 숨겨진 테스트 보드까지 통과
```

### 18.2 숨겨진 테스트 보드

클리어 후, 같은 규칙을 다른 보드에도 적용해본다.  
테스트 보드까지 맞으면 높은 랭크를 준다.

### 18.3 스테이지 에디터

개발자용 또는 유저 제작용으로 확장 가능하다.

```text

색상 추가
규칙 후보 설정
목표 자동 생성
난이도 검사
JSON 내보내기
```

### 18.4 애니메이션 피드백

```text
규칙 적용 방향 표시
```

## 19. 구현 난이도 평가

```text
프로토타입: 낮음
게임다운 UI/UX: 중간
스테이지 에디터: 중간
자동 난이도 평가: 중간~높음
후반 규칙 시스템: 중간~높음
```

핵심 엔진은 어렵지 않다.  
어려운 부분은 좋은 규칙 조합과 좋은 스테이지를 찾는 것이다.

## 20. 최종 방향성

이 게임은 정답을 숨기는 퍼즐이 아니다.  
정답을 처음부터 보여주고, 그 정답이 성립하는 법칙을 플레이어가 설명하게 만드는 게임이다.

핵심 정체성은 다음 문장으로 유지한다.

> 플레이어는 보드를 고치지 않는다.  
> 플레이어는 보드를 만든 규칙을 복원한다.

초기 개발에서는 큰 시스템보다 작은 MVP가 중요하다.

```text
5x5 보드
색 2개
방향 + 거리 + 효과
숫자 누적
10스테이지
Unity 기반 시뮬레이션
```

이 정도만 구현해도 아이디어의 재미를 충분히 검증할 수 있다.
