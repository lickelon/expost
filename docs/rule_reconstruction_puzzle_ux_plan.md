# 규칙 복원 퍼즐 UX 계획

## 방향

현재 프로토타입의 기본 루프는 이미 동작한다.

```text
규칙 선택 -> 시뮬레이션 테스트 -> 목표 보드와 비교 -> 클리어 또는 오답
```

다음 UX 목표는 플레이어가 버튼 조합을 누르는 것이 아니라, 가설을 조립하고 실험한다고 느끼게 만드는 것이다.

핵심 원칙:

- 플레이어는 보드 칸이 아니라 규칙을 수정한다.
- 규칙 효과는 색상자가 남긴 흔적으로 읽혀야 한다.
- 오답 피드백은 다음 행동을 추론할 수 있게 해야 한다.
- UI는 디버그 도구가 아니라 퍼즐 인터페이스처럼 보여야 한다.

## 시뮬레이션 원칙

규칙 적용은 순서와 무관해야 한다.

```text
모든 색상자가 동시에 기여한다
각 칸은 받은 영향을 누적한다
최종 보드는 누적된 기여값으로 계산한다
```

순서 의존성을 만드는 연산은 피한다.

- 덮어쓰기
- 이전 값 지우기
- 현재 값을 두 배로 만들기
- 이미 영향받은 칸 조건
- 중간 결과값 기반 조건

허용 가능한 연산 계열:

- 더하기
- 빼기
- 최댓값
- 최솟값
- OR
- XOR
- 개수 세기

MVP에서는 `전체 영향` 채널과 `더하기` 연산만 사용한다.

## 규칙 카드 UX

각 색은 하나의 규칙 카드로 표현한다.

```text
Red Rule
Target: Cross, 1 Tile
Effect: Influence +1
```

MVP 카드:

```text
[색상 프리뷰] Red Rule
[대상 모양 슬롯] [거리 슬롯] -> [효과 슬롯]
```

확장형:

```text
Red Rule
Target: Cross, 2 Tiles, Empty Cells
Effect: Influence +1
```

나중에 한 색이 여러 효과를 가지면 같은 카드 안에 여러 흔적을 둔다.

```text
Red Rule

Trace 1: Cross, 1 Tile -> Influence +1
Trace 2: Diagonal, 1 Tile -> Influence -1
```

## 효과 블럭

효과는 카드에 장착하는 부품처럼 보여야 한다.

효과 트레이 예시:

```text
[Influence +1] [Influence +2] [Influence -1] [Red +1] [Mark A]
```

기본 조작:

```text
규칙 카드 선택 -> 효과 블럭 클릭 -> 해당 카드에 장착
```

드래그 앤 드롭은 이후 보조 입력으로 추가할 수 있다.

스테이지마다 사용할 수 있는 효과 블럭만 보여줘야 한다. 너무 많은 효과를 한 번에 보여주면 인벤토리처럼 복잡해진다.

추천 진행:

```text
튜토리얼: Influence +1
초반: Influence +1, Influence +2
중반: Influence +1, Influence +2, Influence -1, 색상별 채널
후반: Mark A, OR Mark, XOR Mark, Max 2
```

## 대상 선택

대상 선택은 효과 트레이가 아니라 규칙 카드 안의 슬롯으로 둔다.

기본 슬롯:

```text
모양 + 거리
```

예시:

```text
Cross, 1 Tile
Diagonal, 2 Tiles
Horizontal, Until Wall
```

확장 슬롯:

```text
모양 + 거리 + 조건
```

허용 가능한 조건은 초기 보드 상태만 기준으로 한다.

- 빈 칸
- 색상자 칸 제외
- 벽 앞까지
- 같은 색상자 주변
- 특정 지형
- 보드 가장자리

피해야 할 조건:

- 이미 영향받은 칸
- 숫자가 0보다 큰 칸
- 다른 색이 먼저 닿은 칸
- 현재 결과값이 특정 값인 칸

## 오답 피드백

오답 피드백은 어떤 조정이 필요한지 알려줘야 한다.

칸별 diff:

```text
목표 2, 결과 1 -> +1 필요
목표 0, 결과 1 -> -1 초과
```

요약:

```text
WRONG
8칸 불일치
3칸은 영향 부족
5칸은 영향 초과
```

클리어:

```text
CLEAR
모든 흔적 일치
```

실패 후에는 결과 보드를 계속 확인할 수 있어야 한다. 큰 결과 배너는 짧게만 표시한다.

## 영향 시각화

플레이어는 숫자가 왜 생겼는지 읽을 수 있어야 한다.

규칙 카드 hover:

```text
Red Rule hover -> 빨간 색상자가 영향을 주는 칸 하이라이트
```

효과 블럭 hover:

```text
Influence +1 hover -> 선택된 규칙 기준으로 +1이 들어갈 칸 미리보기
```

칸 hover:

```text
Cell (3, 2)
Total: 2
Red +1
Green +1
```

보드는 단순 숫자 그리드가 아니라 흔적을 읽는 표면이어야 한다.

## 테스트 흐름

`Run`은 `Test`로 바꾸는 편이 좋다.

버튼은 현재 가설을 테스트하고 시뮬레이션 연출을 보여주는 역할이다. 코드를 실행하는 느낌이면 안 된다.

시뮬레이션 중에는 `4/4` 같은 모호한 진행률 대신 상태 문장을 사용한다.

```text
Applying Red...
Applying Blue...
Applying Green...
Comparing...
```

진행 상태와 판정 상태는 분리한다.

```text
Status: Applying Green
Result: Not checked yet
```

완료 후:

```text
Status: Done
Result: WRONG - 8 cells differ
```

## 자동 미리보기

장기적으로는 규칙을 바꿀 때 결과 보드가 즉시 갱신되는 것이 좋다.

```text
규칙 카드 변경 -> 결과 보드 갱신 -> diff 갱신
```

이 구조에서는 `Test`가 유일한 확인 수단이 아니라, 최종 확인과 연출 버튼이 된다.

## 스테이지 제목

스테이지 제목은 정답을 직접 말하면 안 된다.

피해야 할 제목:

```text
Three Color Cross
Diagonal Two
Horizontal Rule
```

추천 제목:

```text
First Trace
Two Sources
Shared Lines
The Middle Two
False Symmetry
Overlapping Clues
```

제목은 정답보다 관찰해야 할 현상을 암시하는 정도가 좋다.

## 튜토리얼 진행

추천 초반 스테이지:

```text
01 색 1개, 모양 찾기
02 색 1개, 거리 찾기
03 색 2개, 겹침 없음
04 색 2개, 한 칸 겹침
05 색 2개, 서로 다른 모양
06 색 3개 등장
```

첫 스테이지가 가르쳐야 할 문장:

```text
각 숫자는 그 칸에 도달한 영향의 개수입니다.
색상자마다 다른 규칙을 찾아 목표 흔적을 재현하세요.
```

## 구현 우선순위

1. 규칙 카드와 효과 슬롯 구조
2. 클릭 장착 방식의 효과 트레이
3. 부족/초과를 보여주는 칸별 diff
4. 규칙 카드 hover 영향 하이라이트
5. 칸 hover 영향 출처 표시
6. 자동 미리보기
7. `Run`을 `Test`로 변경
8. 숫자 진행률을 상태 문장으로 변경

## MVP UX 목표

MVP는 다음을 만족해야 한다.

- 플레이어가 규칙을 조립한다고 느낀다.
- 효과가 장착 가능한 부품처럼 느껴진다.
- 규칙 변경이 보드에 어떤 결과를 만드는지 보인다.
- 오답 결과가 다음 조정을 암시한다.
- 인터페이스가 코딩 도구나 디버그 도구처럼 보이지 않는다.
